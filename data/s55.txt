Naming</h2>

We call a position in the array a "good index"</strong> if starting at that position, we can reach the last index. Otherwise, that index is called a "bad index"</strong>. The problem then reduces to whether or not index 0 is a "good index".</li>
</ul>
Solution</h2>
This is a dynamic programming[1]</sup> question. Usually, solving and fully understanding a dynamic programming problem is a 4 step process:</p>

Start with the recursive backtracking solution</li>
Optimize by using a memoization table (top-down[3]</sup> dynamic programming)</li>
Remove the need for recursion (bottom-up dynamic programming)</li>
Apply final tricks to reduce the time / memory complexity</li>
</ol>
All solutions presented below produce the correct result, but they differ in run time and memory requirements.</p>

<h4 id="approach-1-backtracking-stack-overflow">Approach #1 (Backtracking) [Stack Overflow]</h4>
This is the inefficient solution where we try every single jump pattern that takes us from the first position to the last. We start from the first position and jump to every index that is reachable. We repeat the process until last index is reached. When stuck, backtrack.</p>

            return</span> true</span>;</span>

<h4 id="approach-2-dynamic-programming-top-down-stack-overflow">Approach #2 (Dynamic Programming Top-down) [Stack Overflow]</h4>
Top-down Dynamic Programming can be thought of as optimized backtracking. It relies on the observation that once we determine that a certain index is good / bad, this result will never change. This means that we can store the result and not need to recompute it every time.</p>
Therefore, for each position in the array, we remember whether the index is good or bad. Let's call this array memo</code> and let its values be either one of: GOOD, BAD, UNKNOWN. This technique is called memoization[2]</sup>.</p>
An example of a memoization table for input array nums = [2, 4, 2, 1, 0, 2, 0]</code> can be seen in the diagram below. We write G</strong> for a GOOD</em> position and B</strong> for a BAD</em> one. We can see that we cannot start from indices 2, 3 or 4 and eventually reach last index (6), but we can do that from indices 0, 1, 5 and (trivially) 6.</p>

If it is known then return True</em> / False</em></li>

<h4 id="approach-3-dynamic-programming-bottom-up-time-limit-exceeded">Approach #3 (Dynamic Programming Bottom-up) [Time limit exceeded]</h4>
Top-down to bottom-up conversion is done by eliminating recursion. In practice, this achieves better performance as we no longer have the method stack overhead and might even benefit from some caching. More importantly, this step opens up possibilities for future optimization. The recursion is usually eliminated by trying to reverse the order of the steps from the top-down approach.</p>
The observation to make here is that we only ever jump to the right. This means that if we start from the right of the array, every time we will query a position to our right, that position has already be determined as being GOOD</em> or BAD</em>. This means we don't need to recurse anymore, as we will always hit the memo</code> table.</p>
Java</strong></p>
</span>enum</span> Index</span> {</span>
    GOOD</span>,</span> BAD</span>,</span> UNKNOWN</span>
}</span>

    public</span> boolean</span> canJump</span>(</span>int</span>[]</span> nums</span>)</span> {</span>
        Index</span>[]</span> memo</span> =</span> new</span> Index</span>[</span>nums</span>.</span>length</span>];</span>
        for</span> (</span>int</span> i</span> =</span> 0</span>;</span> i</span> &lt;</span> memo</span>.</span>length</span>;</span> i</span>++)</span> {</span>
            memo</span>[</span>i</span>]</span> =</span> Index</span>.</span>UNKNOWN</span>;</span>

        memo</span>[</span>memo</span>.</span>length</span> -</span> 1</span>]</span> =</span> Index</span>.</span>GOOD</span>;</span>

<h4 id="approach-4-greedy-accepted">Approach #4 (Greedy) [Accepted]</h4>
Once we have our code in the bottom-up state, we can make one final, important observation. From a given position, when we try to see if we can jump to a GOOD</em> position, we only ever use one - the first one (see the break statement). In other words, the left-most one. If we keep track of this left-most GOOD</em> position as a separate variable, we can avoid searching for it in the array. Not only that, but we can stop using the array altogether.</p>
Iterating right-to-left, for each position we check if there is a potential jump that reaches a GOOD</em> index (currPosition + nums[currPosition] &gt;= leftmostGoodIndex</code>). If we can reach a GOOD</em> index, then our position is itself GOOD</em>. Also, this new GOOD</em> position will be the new leftmost GOOD</em> index. Iteration continues until the beginning of the array. If first position is a GOOD</em> index then we can reach the last index from the first position.</p>
To illustrate this scenario, we will use the diagram below, for input array nums = [9, 4, 2, 1, 0, 2, 0]</code>. We write G</strong> for GOOD</em>, B</strong> for BAD</em> and U</strong> for UNKNOWN</em>. Let's assume we have iterated all the way to position 0 and we need to decide if index 0 is GOOD</em>. Since index 1 was determined to be GOOD</em>, it is enough to jump there and then be sure we can eventually reach index 6. It does not matter that nums[0]</code> is big enough to jump all the way to the last index. All we need is one</strong> way.</p>

Conclusion</h2>
The question left unanswered is how should one approach such a question in an interview scenario. I would say "it depends". The perfect solution is cleaner and shorter than all the other versions, but it might not be so straightforward to figure out.</p>
The (recursive) backtracking is the easiest to figure out, so it is worth mentioning it verbally while warming up for the tougher challenge. It might be that your interviewer actually wants to see that solution, but if not, mention that there might be a dynamic programming solution and try to think how could you use a memoization table. If you figure it out and the interviewer wants you to go for the top-down approach, it will not generally be time to think of the bottom-up version, but I would always mention the advantages of this technique as a final thought in the interview.</p>
Most people are stuck when converting from top-down Dynamic Programming (expressed naturally in recursion) to bottom-up. Practicing similar problems will help bridge this gap.</p>
